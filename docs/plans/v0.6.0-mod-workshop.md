# Ship v0.6.0 Mod Workshop foundations

This ExecPlan is a living document. Sections Progress, Surprises &
Discoveries, Decision Log, and Outcomes & Retrospective must be
kept up to date as work proceeds.

## Purpose / Big Picture

`v0.6.0` should make Open Rebellion safe to play longer than a single session and practical to mod without editing source code. At the end of this sprint, fog of war should stop relying on literal fleet presence only, captured characters should be able to escape on normal ticks, saves should survive schema changes and reject mismatched mod sets with clear errors, native builds should expose a working mod manager, and release artifacts should be reproducible for web and macOS packaging.

The easiest way to see this working will be to run the desktop app, start a game, open the new Mods panel, enable or disable a sample mod, save the game, reload it, and verify that the same enabled mod set is restored or that the load is blocked with a specific compatibility message. A second proof point is automated: targeted `cargo test` runs should cover sensor-based fog reveal, per-tick escape checks, save migration, and mod discovery or enable-disable behavior.

## Surprises & Discoveries

- Observation: The mod system is implemented as a library but is not actually used by the running app yet.
  Evidence: `rg -n "ModLoader|ModWatcher" crates` only finds definitions and tests in `crates/rebellion-data/src/mods.rs`; there are no call sites in `crates/rebellion-app/src/main.rs`.

- Observation: The save/load panel exists in the render crate but is not drawn or managed by the main loop.
  Evidence: `crates/rebellion-render/src/panels/save_load.rs` defines `SaveLoadPanelState` and `draw_save_load`, but `crates/rebellion-app/src/main.rs` never constructs `SaveLoadPanelState` and never calls `draw_save_load`.

- Observation: The current save format is version-gated by exact equality and contains no mod compatibility header.
  Evidence: `crates/rebellion-data/src/save.rs` uses `SAVE_VERSION: u32 = 3` and rejects anything where `version != SAVE_VERSION` with `incompatible save version`.

- Observation: The current capture model is not strong enough to support real escape behavior.
  Evidence: `crates/rebellion-core/src/missions.rs` defines `CharacterCaptured { character, captured_by, at_system }`, but `crates/rebellion-app/src/main.rs` currently applies capture by flipping faction booleans and removing the character from fleets. `crates/rebellion-core/src/world/mod.rs` has no explicit captivity field.

- Observation: The README promises broader mod capability than the current code supports.
  Evidence: `README.md` says mods can “Add anything,” but `crates/rebellion-data/src/mods.rs` `ModLoader::apply()` only merge-patches existing serialized arena entries by matching `id`.

## Decision Log

- Decision: Scope `v0.6.0` to mechanics correctness, save robustness, runtime mod usability, and release prep; do not try to finish the remaining 15+ story beats here.
  Rationale: The story event backlog is content-heavy and open-ended. This phase is better used to harden the engine and make future event work easier to ship and test.
  Date/Author: 2026-03-16 / Codex

- Decision: Treat the roadmap item “probe droids / sensor range” as a fog-of-war visibility radius pass, not as a full consumable probe-droid inventory system.
  Rationale: The codebase already has fleet detection stats and a fog system. It does not have a dedicated probe-droid entity, inventory, or mission flow. A sensor-radius implementation closes the correctness gap without inventing a larger subsystem mid-sprint.
  Date/Author: 2026-03-16 / Codex

- Decision: Add explicit captivity state before wiring `ESCAPETB`.
  Rationale: Escape checks need a stable representation of “this character is captured, by whom, and where.” The current faction-flip shortcut cannot support per-tick escape, rescue, or save compatibility cleanly.
  Date/Author: 2026-03-16 / Codex

- Decision: Include the `ResearchSystem::advance()` contract fix in this sprint, but defer `BuildableKind` class-arena type safety.
  Rationale: The research contract is a contained architectural cleanup that reinforces the project-wide stateless `advance()` rule. `BuildableKind` requires wider data-model expansion across troops and facilities and is more suited to a later dedicated refactor.
  Date/Author: 2026-03-16 / Codex

- Decision: Ship a basic Mod Manager in `v0.6.0`: discover, inspect, enable, disable, reload, and show dependency or error state. Do not invent a per-mod settings schema yet.
  Rationale: Current manifests have no configuration schema. “Configure mods” should mean enable-disable plus visibility into manifest metadata for this sprint; deeper settings need a separate design.
  Date/Author: 2026-03-16 / Codex

- Decision: Limit distribution work to reproducible artifacts and repo-owned metadata.
  Rationale: This sprint should produce packaging scripts, vendored web dependencies, a Homebrew formula template, and release notes scaffolding. Actually publishing to itch.io, Homebrew taps, or GitHub Releases can happen after the artifacts are proven locally.
  Date/Author: 2026-03-16 / Codex

## Outcomes & Retrospective

Pending implementation. If this plan is completed as written, `v0.6.0` will not mean “all modding dreams are done.” It will mean the engine can load and validate mods at runtime, save files know which mod set they belong to, the fog and escape rules are closer to the original game’s intent, and release packaging stops depending on ad hoc local steps.

## Context and Orientation

Open Rebellion is a five-crate Rust workspace. `crates/rebellion-core` contains pure simulation types and logic with no file system access. `crates/rebellion-data` is the loading and serialization layer. `crates/rebellion-render` owns macroquad and egui drawing. `crates/rebellion-app/src/main.rs` is the orchestration layer that constructs simulation state, advances every system, and applies all world mutations.

The project uses an `advance()` pattern. In this repository that means a simulation system receives the current state and tick input, computes result events, and does not directly perform file access. This matters because migration, replay, and testing all get harder when systems both mutate internal state and also ask callers to apply overlapping effects.

A “save migration” in this document means a deliberate code path that converts an older save format into the newest in-memory `SaveState`, instead of simply rejecting any older version. A “manifest” means a mod’s `mod.toml` file, which currently contains metadata and dependency rules. “Hot reload” means re-reading the mods directory while the game is already running on native platforms. “Sensor radius” means revealing systems near a friendly fleet based on a numeric visibility range, instead of revealing only the exact system containing the fleet.

Key files and why they matter:

- `agent_docs/roadmap.md`: defines the remaining roadmap items and the intended `Mod Workshop` milestone.
- `docs/CHANGELOG.md`: shows `v0.5.0` as roughly 90% complete and confirms save format version `3`.
- `crates/rebellion-core/src/fog.rs`: current fog logic; explicitly says sensor or probe-droid range is deferred.
- `crates/rebellion-core/src/missions.rs`: mission resolution, `CharacterCaptured`, `CharacterEscaped`, decoy logic, and the current `check_escapes()` stub.
- `crates/rebellion-core/src/world/mod.rs`: `Character`, `Fleet`, and `GameWorld` runtime types; this is where captivity or sensor metadata must become durable.
- `crates/rebellion-core/src/research.rs`: current research advancement logic that still deviates from the project-wide `advance()` contract.
- `crates/rebellion-data/src/save.rs`: current save header and `SaveState`; uses exact-version equality and has no mod metadata.
- `crates/rebellion-data/src/mods.rs`: mod manifest parsing, dependency sort, RFC 7396 merge patch application, and native file watching.
- `crates/rebellion-data/src/lib.rs`: base-game data loading; currently the obvious place to centralize “load base data, then apply selected mods.”
- `crates/rebellion-render/src/panels/mod.rs`: panel registry and `PanelAction`; a new Mods panel should fit the existing borrow-safe UI pattern here.
- `crates/rebellion-render/src/panels/save_load.rs`: existing save/load modal; the mod manager should follow the same style and the save/load panel itself still needs actual app integration.
- `crates/rebellion-app/src/main.rs`: the real control point for system ordering, mod reload, save/load calls, and panel orchestration.
- `scripts/build-wasm.sh`: current web packaging script; it still downloads `web/gl.js` from the network on first build, which is not acceptable for reproducible release packaging.
- `README.md`: must be corrected where it over-promises current mod capability and updated with actual mod-manager and release instructions.

## Plan of Work

Start with correctness because everything later depends on stable runtime semantics. In `crates/rebellion-core/src/world/mod.rs`, add explicit capture metadata for characters or a dedicated captivity record type, then update `crates/rebellion-core/src/missions.rs` so capture, rescue, and escape all operate on that explicit state instead of silent faction flips. Wire per-tick escape checks into the main loop in `crates/rebellion-app/src/main.rs` and add focused tests in `crates/rebellion-core/src/missions.rs` for capture persistence, rescue cleanup, and `ESCAPETB` outcomes. In the same correctness pass, extend `crates/rebellion-core/src/fog.rs` so fleets reveal a radius derived from existing detection-related data rather than only their exact location. This should remain deterministic and testable. Finish the tranche by normalizing `crates/rebellion-core/src/research.rs` to the same “results out, caller applies” contract used elsewhere.

Then harden save and load. Replace the bare header in `crates/rebellion-data/src/save.rs` with a small save envelope that includes schema version, minimum readable version, enabled mod manifests, and a deterministic mod-set hash. The load path should deserialize older versions through named migration functions, not through an `if version != current` guard. Store fixture saves under a repository path such as `crates/rebellion-data/tests/fixtures/` and make at least one real fixture for the current `v0.5.0` save format so the first migration is not hypothetical. Because the app currently does not use the save/load panel, wire `crates/rebellion-render/src/panels/save_load.rs` into `crates/rebellion-app/src/main.rs` as part of the same tranche; otherwise the new robustness stays unreachable.

Next, turn the mod system from a library into a feature players can use. Create a mod-runtime layer in `crates/rebellion-data/src/mods.rs` or a nearby module that reads discovered manifests, applies only enabled mods, persists the enabled set to a small config file, and surfaces dependency or version problems as structured values instead of `eprintln!` only. Add `crates/rebellion-render/src/panels/mod_manager.rs`, register it in `crates/rebellion-render/src/panels/mod.rs`, and add new `PanelAction` variants so `crates/rebellion-app/src/main.rs` can open the panel, toggle mods, reload data, and rebuild the live world safely. The hot-reload watcher should work only on native, but the UI must clearly explain that browser builds do not support filesystem watching.

Finish by writing the documentation and packaging glue that make the feature set usable. Add a real example mod directory under `mods/examples/` or another clearly documented sample location. Update `README.md` and `agent_docs/modding.md` so they describe the actual patch-only capability, the new enabled-mod flow, and the save compatibility rules. Vendor `web/gl.js` into the repository and stop `scripts/build-wasm.sh` from fetching network content during builds. Add packaging scripts and metadata for a browser zip, a macOS archive, and a Homebrew formula template under a repository-owned path such as `packaging/` or `dist/`. The goal is not store publication in this sprint; the goal is that one person can run the scripts twice and get the same artifact layout both times.

## Validation and Acceptance

Acceptance for this sprint is behavioral, not just compile-level.

- Fog correctness: a deterministic unit test in `crates/rebellion-core/src/fog.rs` proves that a fleet with sensor strength reveals nearby systems without occupying them, and a live run shows reveal messages before fleet arrival when the radius rule applies.
- Escape correctness: a deterministic unit test in `crates/rebellion-core/src/missions.rs` proves that a captured character can remain captive across ticks, escape via `ESCAPETB`, and stop being eligible for rescue or repeated escape once freed.
- Research contract: `crates/rebellion-core/src/research.rs` no longer mutates state and returns overlapping world-change events at the same time. The app layer should apply research unlock results explicitly, exactly once.
- Save compatibility: a version-3 fixture loads through the migration path, and a save created with one enabled mod set fails to load under a different mod set with a precise error that names the mismatch.
- Save UX: the in-app save/load panel is reachable, populated from actual slot metadata, and surfaces failures in the panel instead of only printing to stderr.
- Mod manager UX: the in-app Mods panel lists discovered mods, shows version and dependency information, allows enable-disable, persists that choice, and lets the player reload without restarting the app.
- Documentation: `README.md`, `agent_docs/modding.md`, and the example mod all agree on what the current mod system can really do. They must not claim additive entity creation unless code for additive creation exists in this sprint.
- Packaging: the web build succeeds with no network fetches, and the release packaging scripts generate deterministic file names and a documented artifact layout suitable for itch.io, GitHub Releases, and a future Homebrew tap.

## Idempotence and Recovery

This plan is intentionally safe to repeat. Test commands are read-only. Save migrations should be append-only functions that never rewrite older fixtures in place. Running the packaging scripts multiple times should overwrite a dedicated output directory such as `dist/` without touching source assets. Mod enable-disable state should live in a small config file that can be deleted and regenerated safely.

If a migration step goes wrong, recovery is to delete only the newly added save fixture or generated output, not to edit historical fixtures in place. If runtime mod loading causes instability, recovery is to clear the enabled-mod config file and restart with base data only. If a release packaging change breaks the web build, restore the vendored `web/gl.js` and keep the old script behavior in Git history for comparison, but do not reintroduce network fetches as the final state.

## Interfaces and Dependencies

The following interfaces or equivalents should exist when `v0.6.0` is complete:

- `crates/rebellion-core/src/world/mod.rs` must represent captivity explicitly, either on `Character` or in a new serializable runtime structure owned by mission state.
- `crates/rebellion-core/src/missions.rs` must provide a deterministic per-tick escape path. Whether this remains `MissionSystem::check_escapes(...)` or becomes part of `MissionSystem::advance(...)`, the contract must consume caller-supplied rolls and return explicit escape effects.
- `crates/rebellion-core/src/fog.rs` must compute visibility from more than exact fleet location. The implementation must remain pure and testable inside `rebellion-core`.
- `crates/rebellion-core/src/research.rs` must conform to the project’s stateless `advance()` expectation: state evolves in one place, and caller-visible results are applied once.
- `crates/rebellion-data/src/save.rs` must define a versioned save envelope with mod metadata and a migration entry point, not only a bare `SAVE_VERSION` equality check.
- `crates/rebellion-data/src/mods.rs` must expose structured runtime data for discovered mods, enabled mods, dependency errors, and native file-watch changes. Silent `eprintln!` output is not enough for UI-driven workflows.
- `crates/rebellion-render/src/panels/mod_manager.rs` must follow the existing panel pattern used by `crates/rebellion-render/src/panels/save_load.rs`: render-only UI, `PanelAction` out, world mutation in the app layer.
- `crates/rebellion-app/src/main.rs` must become the single orchestrator for base data load, selected-mod application, save/load calls, reload flows, and panel actions.
- No new dependency should violate the current layering rule that `rebellion-core` stays pure and contains zero file system or UI code.

Expected library usage:

- Existing crates already in use: `serde`, `serde_json`, `bincode`, `toml`, `semver`, `notify`, `macroquad`, `egui-macroquad`.
- No new dependency is required for the core plan unless a checksum crate is chosen for deterministic mod-set hashing. If one is added, keep it out of `rebellion-core` unless the hash must be computed from pure data only.
