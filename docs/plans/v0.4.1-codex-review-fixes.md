# v0.4.1: Codex Review Integration Fixes

## Context

GPT-5.4 reviewed the entire Open Rebellion workspace and found 4 critical architectural gaps and 5 important issues. These represent the difference between "code exists" and "systems are fully integrated." All 23 Knesset Tiamat tasks are complete, but the new systems (blockade, uprising, Death Star, research, Jedi, victory) are not wired into the main loop or save/load.

## Critical Fixes (4)

### Fix 1: Fleet arrivals must update System.fleets

**File**: `crates/rebellion-app/src/main.rs:226-228`
**Problem**: When a fleet arrives at a new system, `Fleet.location` is updated but the fleet is never removed from `origin_system.fleets` or added to `destination_system.fleets`. Combat, fog, blockade, and victory all query `System.fleets` and see stale positions.
**Fix**: In the arrival handling block, after `fleet.location = arrival.system`:
```rust
// Remove from origin system's fleet list
if let Some(origin_sys) = world.systems.get_mut(arrival.origin) {
    origin_sys.fleets.retain(|&k| k != arrival.fleet);
}
// Add to destination system's fleet list
if let Some(dest_sys) = world.systems.get_mut(arrival.system) {
    if !dest_sys.fleets.contains(&arrival.fleet) {
        dest_sys.fleets.push(arrival.fleet);
    }
}
```
**Requires**: `ArrivalEvent` needs an `origin: SystemKey` field added to `movement.rs`.

### Fix 2: Wire new systems into main loop + save/load

**Files**: `crates/rebellion-app/src/main.rs`, `crates/rebellion-data/src/save.rs`
**Problem**: BlockadeState, UprisingState, DeathStarState, ResearchState, JediState, VictoryState are defined in rebellion-core but never instantiated in main.rs or serialized in SaveState.
**Fix**:
1. In `main.rs`, after existing state initialization:
```rust
let mut blockade_state = BlockadeState::new();
let mut uprising_state = UprisingState::new();
let mut death_star_state = DeathStarState::new();
let mut research_state = ResearchState::new();
let mut jedi_state = JediState::new();
let mut victory_state = VictoryState::new(/* alliance_hq, empire_hq */);
```
2. In the tick loop, after AI actions:
```rust
// Blockade
let blockade_events = BlockadeSystem::advance(&mut blockade_state, &world, &tick_events);
// Uprising
let uprising_events = UprisingSystem::advance(&mut uprising_state, &world, &tick_events, &uprising_rolls);
// Death Star
let ds_events = DeathStarSystem::advance(&mut death_star_state, &world, &tick_events);
// Research
let research_events = ResearchSystem::advance(&mut research_state, &world, &tick_events);
// Jedi
let jedi_events = JediSystem::advance(&mut jedi_state, &world, &tick_events, &jedi_rolls);
// Victory check (every tick)
if let Some(outcome) = VictorySystem::check(&victory_state, &world) { ... }
```
3. In `save.rs`, add all 6 new states to `SaveState` struct.

### Fix 3: Implement missing mission effect world mutations

**File**: `crates/rebellion-app/src/main.rs` (apply_mission_result)
**Problem**: AI dispatches sabotage/assassination/espionage but `apply_mission_result` drops non-popularity effects.
**Fix**: Add match arms for new MissionEffect variants:
- `FacilitySabotaged { system, facility }` → remove facility from system
- `CharacterKilled { character, .. }` → remove character from world
- `CharacterCaptured { character, captured_by, .. }` → transfer character faction
- `CharacterRescued { character, .. }` → return character to original faction
- `UprisingStarted { system, .. }` → trigger uprising via UprisingState

### Fix 4: BuildableKind type safety for non-ship types

**File**: `crates/rebellion-core/src/manufacturing.rs`
**Problem**: `BuildableKind::Troop(TroopKey)` etc. use instance keys not class keys.
**Fix**: For v0.4.1, add a comment documenting the limitation. Full fix (class arenas for troops/facilities) is a v0.5.0 concern — requires DAT layer expansion similar to what Karme did for CapitalShipClass.

## Important Fixes (5)

### Fix 5: Jedi XP persistence

**File**: `crates/rebellion-core/src/jedi.rs`
**Problem**: `JediSystem::advance` computes `new_xp` but only emits events on tier changes. XP below threshold never persists.
**Fix**: Either emit an `XpGained { character, new_xp }` event every tick, or write `force_experience` directly in advance() (consistent with how other systems work).

### Fix 6: Event system multi-tick safety

**File**: `crates/rebellion-core/src/events.rs`
**Problem**: `EventSystem::advance` only checks `tick_events.last()`. `TickReached { tick: N }` can be skipped if a frame spans multiple days.
**Fix**: Iterate over each `TickEvent` individually for tick-sensitive conditions.

### Fix 7: Research system advance() contract

**File**: `crates/rebellion-core/src/research.rs`
**Problem**: `ResearchSystem::advance` both mutates state AND returns events, breaking the advance() pattern.
**Fix**: Choose one: pure results (return completed projects, caller applies) or stateful (advance internally, return notifications only).

### Fix 8: Thread blockade into manufacturing

**File**: `crates/rebellion-core/src/manufacturing.rs` or `crates/rebellion-app/src/main.rs`
**Problem**: `BlockadeState::is_blockaded()` is never checked before manufacturing advancement.
**Fix**: In main loop, skip manufacturing for blockaded systems: `if !blockade_state.is_blockaded(system) { ManufacturingSystem::advance(...) }`

### Fix 9: Per-tick event processing

**Files**: Multiple systems (manufacturing, missions, movement, death_star, uprising, blockade)
**Problem**: Several systems collapse multi-tick frames to a single `last_tick` timestamp.
**Fix**: Process `tick_events` one-at-a-time in systems where ordering matters. Low priority — only matters at very low frame rates.
